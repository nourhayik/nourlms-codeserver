# AI API — developer guide

This document describes how to call the **OpenAI-compatible** backend used for LLM features. It is written for engineers integrating or debugging AI calls; it does not describe application business logic.

---

## Base URL

| Environment | Base URL |
|-------------|----------|
| Production / default | `https://ai.nourlms.com/v1` |

All paths below are relative to this base (no trailing slash on the base in client config is fine; SDKs normalize it).

**Full chat endpoint:** `https://ai.nourlms.com/v1/chat/completions`

Official OpenAI API reference (same request/response shape): [Chat completions](https://platform.openai.com/docs/api-reference/chat/create).

---

## Authentication

Use a **Bearer token** in the `Authorization` header.

```http
Authorization: Bearer reMoT-sKMqhgjSxWOkcQ7RfaeaZn7sMxnLs30AWT3uw
```

- **Header name:** `Authorization`
- **Value format:** `Bearer ` + API key (single space after `Bearer`)
- **Content-Type:** `application/json` for POST bodies

### Team API key (copy for env / other projects)

```
reMoT-sKMqhgjSxWOkcQ7RfaeaZn7sMxnLs30AWT3uw
```

Use this value for `Authorization: Bearer …`, for `AI_API_KEY` / `OPENAI_API_KEY`, and anywhere examples below say `<YOUR_API_KEY>`. Anyone with this string can consume your AI quota—keep the repo private or rotate the key if this file is ever exposed.

**Suggested environment variables (for your own apps):**

| Variable | Purpose |
|----------|---------|
| `AI_API_BASE_URL` | Optional override; default `https://ai.nourlms.com/v1` |
| `AI_API_KEY` or `OPENAI_API_KEY` | Same key as above |

## Chat completions (primary API)

### `POST /chat/completions`

Request body (minimal fields used in our stack):

| Field | Type | Description |
|-------|------|---------------|
| `model` | string | e.g. `gpt-4o-mini` |
| `messages` | array | OpenAI-style chat messages (`role`: `system` \| `user` \| `assistant`, `content`: string) |
| `temperature` | number | Optional; e.g. `0.3` for more deterministic output |
| `response_format` | object | Optional; `{ "type": "json_object" }` to constrain replies to JSON |

### Example: `curl`

```bash
curl -sS https://ai.nourlms.com/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer reMoT-sKMqhgjSxWOkcQ7RfaeaZn7sMxnLs30AWT3uw" \
  -d '{
    "model": "gpt-4o-mini",
    "temperature": 0.3,
    "response_format": { "type": "json_object" },
    "messages": [
      {
        "role": "system",
        "content": "You only reply with valid JSON."
      },
      {
        "role": "user",
        "content": "{\"task\":\"Return a JSON object with one key hello and value world\"}"
      }
    ]
  }'
```

### Example: Node.js (`openai` package)

```bash
npm install openai
```

```javascript
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.AI_API_KEY,
  baseURL: process.env.AI_API_BASE_URL ?? "https://ai.nourlms.com/v1",
});

const completion = await client.chat.completions.create({
  model: "gpt-4o-mini",
  temperature: 0.3,
  response_format: { type: "json_object" },
  messages: [
    { role: "system", content: "You only reply with valid JSON." },
    { role: "user", content: "..." },
  ],
});

const text = completion.choices[0]?.message?.content ?? "";
```

### Example: Python (`openai` library)

```python
from openai import OpenAI
import os

client = OpenAI(
    api_key=os.environ["AI_API_KEY"],
    base_url=os.environ.get("AI_API_BASE_URL", "https://ai.nourlms.com/v1"),
)

completion = client.chat.completions.create(
    model="gpt-4o-mini",
    temperature=0.3,
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": "You only reply with valid JSON."},
        {"role": "user", "content": "..."},
    ],
)

text = completion.choices[0].message.content or ""
```

## Successful response (shape)

Responses follow the OpenAI chat completions schema. The part you usually read:

- `choices[0].message.content` — string (often JSON text when `response_format` is `json_object`)

Parse `content` as JSON in your app when you request JSON output.

---

## Errors

Typical HTTP status codes:

| Code | Meaning |
|------|---------|
| `401` | Missing or invalid `Authorization` / API key |
| `429` | Rate limited — back off and retry |
| `5xx` | Gateway or upstream error — retry with exponential backoff |

Error bodies are usually JSON with an `error` object (OpenAI-style). Log the full response (without logging the API key) when debugging.

---

## Operational checklist for new developers

1. Copy the team API key from the [Authentication](#authentication) section above.
2. Set `AI_API_KEY` (or equivalent) in `.env` / your host secrets for each app; avoid pasting the key into public issue trackers or client-side code.
3. Point clients at `https://ai.nourlms.com/v1` unless told otherwise.
4. Use `POST /chat/completions` with `model: gpt-4o-mini` unless your use case requires another model.
5. For structured output, set `response_format: { type: "json_object" }` and parse `choices[0].message.content`.

## Security reminders

- This document and application code contain the same key—treat the repository as **confidential**; rotate the key if the repo or docs leak.
- Restrict who can create or view keys on the gateway side.
- Prefer server-side calls to the AI API so keys are not exposed to browsers.
